# AGENTS.md

Guidelines for agentic coding agents working in this repository.

## Project Overview

This is a personal blog/notes site built with **Jekyll 4.3.2**, using the
`startbootstrap-clean-blog-jekyll` remote theme. Posts are written in Markdown
and rendered with the Kramdown parser. The site is deployed to GitHub Pages at
`https://maleckidd.github.io`.

No Cursor rules (`.cursor/rules/`, `.cursorrules`) or Copilot instructions
(`.github/copilot-instructions.md`) exist in this repository.

---

## Build & Serve Commands

All commands should be run from the repository root with Bundler to ensure the
correct gem versions are used.

| Command | Purpose |
|---|---|
| `bundle install` | Install all gem dependencies |
| `bundle exec jekyll serve` | Build site and serve locally at `http://localhost:4000` (watch mode) |
| `bundle exec jekyll serve --livereload` | Same as above with live reload |
| `bundle exec jekyll build` | Build the site to `_site/` without serving |
| `bundle exec jekyll build --drafts` | Build including posts in `_drafts/` |
| `bundle exec jekyll doctor` | Check configuration and flag known issues |

Jekyll has **no automated test suite** in this repo. There is no `npm`,
`yarn`, `pytest`, `rspec`, or similar test runner. "Testing" is done by
building and visually inspecting the local server output.

### Quick sanity check (build without serving)

```sh
bundle exec jekyll build 2>&1
```

A successful build exits with code `0` and outputs to `_site/`. Any Liquid
template errors or broken `include` references will surface here.

## Front Matter Conventions

Every post in `_posts/` must include YAML front matter. Required and optional
fields:

```yaml
---
layout: post          # required — always "post" for blog entries
title:  "Post Title"  # required — full display title (quoted string)
subtitle: "Subtitle"  # optional — shown below title in header
date: YYYY-MM-DD HH:MM:SS +0200  # required — must match filename date
categories: CategoryName  # required — single category word (e.g. Cypress, Security)
background: 'https://...'  # optional — URL to header background image
author: "Name"        # optional — falls back to site.author in _config.yml
---
```

### Post filename format

```
YYYY-MM-DD-kebab-case-slug.markdown
```

- Date in the filename must match the `date` field in front matter.
- Use `.markdown` extension (not `.md`) — consistent with existing posts.
- Slug is lowercase, words separated by hyphens.

## Markdown & Content Style

- Markdown flavor: **Kramdown** (`input: Kramdown` in `_config.yml`).
- Syntax highlighting: **Rouge** highlighter.
- Wrap code blocks in `{% highlight language %}...{% endhighlight %}` Liquid
  tags rather than triple backtick fences where syntax highlighting is desired.
  Wrap in a `<div style="background-color:lightgrey;">` for the established
  visual style:

  ```liquid
  <div style="background-color:lightgrey;">
  {% highlight javascript %}
  // code here
  {% endhighlight %}
  </div>
  ```

- Use `**\`keyword\`**` (bold + inline code) for section headings within post
  body text — this is the established pattern in existing posts.
- Add a `<br>` after each bold section heading line to preserve spacing.
- Use `<b>...</b>` for inline bold within HTML-heavy sections.
- Numbered and bulleted lists use standard Markdown; `\t-\t` (tab-dash-tab) is
  used for indented bullet items in some posts — prefer standard `  -` instead
  for new content.
- Prefer `&middot;` for mid-dots, `&larr;`/`&rarr;` for arrows in HTML
  fragments.

## Liquid / Layout Guidelines

- Layouts live in `_layouts/` and use Liquid templating.
- `default.html` is the base layout; all other layouts extend it via
  `layout: default` in their own front matter.
- Jekyll `include` files (e.g. `head.html`, `footer.html`, `scripts.html`,
  `google-analytics.html`, `read_time.html`) are provided by the remote theme —
  do not duplicate them locally unless you need to override them.
- Use `{{ site.baseurl | prepend: ... }}` / `| relative_url` filters when
  constructing internal URLs. The existing pattern uses:
  ```liquid
  {{ post.url | prepend: site.baseurl | replace: '//', '/' }}
  ```
- Conditional background image pattern (used in `home.html` and `post.html`):
  ```liquid
  {% if page.background %}
  <header class="masthead" style="background-image: url('{{ page.background }}')">
  {% else %}
  <header class="masthead">
  {% endif %}
  ```

## Configuration (`_config.yml`)

- **Do not** add per-environment config overrides without documenting them here.
- The `remote_theme` key controls the theme. Override theme files locally by
  placing the file at the same relative path in the repo root.
- Plugins are declared under `plugins:` — any new plugin also needs a
  corresponding `gem` entry in `Gemfile` inside the `:jekyll_plugins` group.
- `kramdown.input: Kramdown` must stay set; changing it will affect how
  Markdown is parsed across all posts.

## Images

- Store post background images in `img/posts/` (e.g. `01.JPG`, `02.JPG`).
- Reference them via the raw GitHub URL pattern used in existing posts:
  ```
  https://raw.githubusercontent.com/Maleckidd/Maleckidd.github.io/gh-pages/img/posts/NN.JPG
  ```
- Image files in `img/` are committed directly to the repo.

## Common Pitfalls

- Running `jekyll serve` without `bundle exec` may use a system-installed
  Jekyll version that differs from the pinned `4.3.2` in `Gemfile`.
- The `feed.xml` is auto-generated by the `jekyll-feed` plugin — do not edit
  it manually.
- Posts dated in the future will not appear by default; pass `--future` to
  `jekyll serve`/`build` to include them.
- The `_site/` directory is the build output and should never be committed
  (add to `.gitignore` if not already present).
